# 企业信息与个人资质查询服务平台 项目全细节复盘（含服务器架构）
本文是个人负责的 企业信息与个人资质查询服务APP+CRM 全细节复盘，涵盖服务器架构、业务逻辑、技术实现、核心难点、踩坑记录等所有细节，用于个人技术沉淀与分享，也希望能给同行提供参考。

## 一、项目概述
### 1.1 项目背景与目标
项目背景：为企业搭建工商信息与个人背景查询类App，配套客户管理（CRM）及经纪人分润结算功能，核心聚焦工商信息查询、个人背景核验核心需求，解决用户查询便捷性、数据准确性及经纪人分润核算繁琐的痛点，替代原有手动查询、人工分润模式，提升业务处理效率、减少人工误差，实现查询服务、客户管理与分润结算的规范化、自动化。
- 核心目标：1.实现工商信息、个人背景的便捷查询，保障查询数据精准、响应高效；2.实现客户从注册、层级绑定到查询订单转化的全流程闭环管理；3.实现查询订单金额分层分润、自动计算、明细留存，确保分润精准可追溯；4.支持系统配置灵活调整（如分润比例、公司分成规则、层级限制等），适配业务迭代需求；5.保障系统高可用、数据安全，支撑日常查询业务7×24小时稳定运行。

### 1.2 项目规模
- 项目规模：后端仅本人负责，搭配1名前端开发，无额外开发人力支撑；系统支持日均查询订单1000+，客户数10000+，分润计算峰值QPS 50+，属于中小规模业务场景。考虑到项目人手有限，为降低开发与后期维护成本，避免复杂架构带来的运维压力，最终采用单体架构，可高效支撑业务运转且便于日常维护。

### 1.3 个人职责
作为唯一后端开发，全程主导后端全链路工作，具体职责包括：负责单体架构设计、数据库设计、接口设计；核心模块开发与优化、重点负责工商/个人背景查询接口对接、分润计算逻辑开发、客户管理、系统配置模块开发；服务器的选购、部署、日常运维及故障排查，保证服务器稳定运行；对接微信、支付宝原生支付及拉卡拉聚合支付，完成支付功能的开发、调试与上线，保证支付流程顺畅合规；对接三方工商、个人背景数据源接口，完成数据交互调试与优化，确保查询数据准确、响应高效；编写接口文档、单元测试，配合前端完成接口联调测试，处理项目开发、测试已经线上运行中的各种技术问题，全程支撑项目从需求落地到上线运维的全流程，兼顾开发、运维、对接等多方面工作，确保项目顺利上线并稳定运行。

## 二、服务器架构
说明：结合项目中小规模业务属性及人手有限的实际情况，采用「单体架构」，兼顾系统稳定性、开发效率与维护成本，无需复杂运维，本人可独立完成部署与维护，以下是从服务器到应用部署的完整架构细节。

### 2.1 服务器硬件配置（线上环境）
采用腾讯云服务器，按环境部署，确保开发、生产环境隔离，避免互相干扰，同时控制服务器成本，适配人手少、易维护的需求，具体配置如下：

| 开发/测试环境 | 1 | 2核/8G | 80G SSD | 开发测试，开发设施搭建：gitea、nexus repository |
| 生产环境 | 1 | 4核/8G | 50G SSD | 生产环境 |

### 2.2 部署架构
开发环境：
- nginx：监听80/443端口、静态资源缓存，提升请求响应速度。
- 测试应用：验证业务流程是否正常，测试接口响应速度。
- 数据库与缓存。

- nexus repository：私有依赖仓库，主要用于上传三方sdk依赖。

生产环境：

- Redis：使用docker部署。

- Fluent-bit：使用docker部署，应用日志收集。

### 2.3 应用架构
系统整体采用单体架构，未进行微服务拆分，但按业务模块划分（如查询模块、支付模块、客户管理模块等），每个业务模块内部均遵循以下5层架构设计，层层依赖、职责明确，既保证模块内的结构清晰，又兼顾单体架构的开发效率与维护便捷性，适配人手少的运维场景。

- 数据层：采用MySQL数据库（主从复制，保障数据安全）、Redis缓存；负责核心业务数据（查询订单、客户信息、分润明细等）存储、高频查询数据缓存加速、数据定时备份，确保查询响应高效、数据不丢失。
- 持久层：MyBatis-Plus + Hutool；负责数据库CRUD操作、数据转换、分页查询、条件构造，简化持久层开发，提升本人开发效率，减少重复代码。
- 业务层：核心业务逻辑实现，重点包括工商/个人背景查询逻辑、分润计算逻辑、客户层级管理、支付接口对接、第三方数据源对接等，同时负责事务控制、业务规则校验，是系统的核心业务处理层。
- 控制层：SpringMVC + Restful API；负责接收前端请求、请求参数校验、返回响应结果、接口异常统一处理，对外提供规范的接口服务，支撑前端查询、支付、分润等功能展示。
- 接入层：Nginx；负责负载均衡、请求转发、静态资源代理、跨域配置、请求限流，保障系统接入的稳定性与安全性，同时降低本人运维压力。

### 2.4 网络架构与安全配置
- 网络配置：服务器之间通过内⽹通信，避免外网直接访问核心服务；外网仅开放Nginx的80/443端口，最大限度降低核心服务暴露风险，对内仅IP白名单开放22、3306、6379端口，同时便于本人维护。
- 安全配置：1.配置文件敏感信息采用Jasypt加密存储，避免明文泄露；2.接口请求加签处理，防止非法请求、接口篡改，保障查询数据与支付数据安全；3.跨域配置明确指定允许访问的前端域名，不使用通配符*，避免安全风险；4.服务器配置防火墙，仅开放必要端口（80、443、数据库内网端口等），禁止无关端口访问；5.第三方数据源接口数据及敏感数据采用AES加密存储，保障查询数据的安全性与合规性。

## 三、项目核心业务细节
### 3.1.1 工商/个人背景查询流程
1. 触发条件：用户完成查询订单支付后，支付接口回调后端查询逻辑。
2. 前置校验：后端校验用户身份合法性（登录状态、权限、三要素验证）、查询参数完整性（关键词非空），校验通过后进入查询流程，校验失败则返回对应提示。
3. 数据源对接：后端调用第三方合规数据源接口，传入查询参数，获取工商/个人背景查询数据（如企业工商信息、资质信息等）。
4. 数据处理与返回：后端对第三方返回的数据进行清洗、格式化，过滤无效信息，封装成统一格式，返回给前端展示；同时保存查询记录（关联用户ID、查询关键词、查询时间、查询结果），便于用户后续查看。
5. 异常处理：若第三方数据源接口调用失败、查询无结果，后端返回明确异常提示，同时记录异常日志，便于本人排查问题。

### 3.1.2 分润计算流程
1. 触发条件：用户完成查询订单支付后，支付接口回调后端分润逻辑，传入订单ID、订单金额、分享人ID（distributorId），触发分润计算。
2. 前置查询：① 根据订单ID查询订单详情（order表），校验订单支付状态、金额合法性；② 根据分享人ID递归查询客户层级关系（从分享人开始，向上查找上级，直到员工/无上级）；③ 从sys_config表查询分润比例、公司分成规则等配置参数。
3. 分润计算：第一步，按订单金额分层计算公司分成；第二步，计算可分润总额（订单金额 - 公司分成）；第三步，按客户层级分配分润金额（一级、二级、三级经纪人按比例分配）；第四步，处理员工直推特殊分润逻辑。
4. 结果存储：生成分润明细（dist_profit表），详细记录每个分润人的ID、分润层级、分润比例、分润金额、关联订单ID、计算时间，确保分润明细可追溯、可对账。

## 3.2 核心业务规则
### 3.2.1 查询业务规则
- 查询权限：仅登录用户可发起查询，未登录用户需跳转至登录页面，避免非法查询；
- 查询频率限制：单个用户单日最多发起10次查询，防止恶意查询、过度占用数据源资源；
- 数据准确性：第三方数据源接口调用失败时，自动重试3次（每次间隔5秒），重试失败则返回查询失败提示；对于任一接口失败，视为报告查询失败，查询失败的报告将在10分钟后再重新进行查询，重查最多2次。
- 查询记录留存：用户所有查询记录（关键词、查询时间、查询结果、订单ID）留存6个月，便于用户追溯、业务统计。

# 五、项目踩坑
## SDK使用的JDK版本问题
- 拉卡拉使用的Java8，HttpServlet接口包名前缀是javax，与我应用的java21版本对不上。使用jakartaee-migration可将javax迁移到jakarta。

- 拉卡拉支付回调在请求头 Authorization 中携带了自定义信息，并非标准 JWT 格式，
与我的 JWT 解析工具直接冲突，导致解析失败。
Spring Security 配置的白名单接口（permitAll）仅代表接口无需登录即可访问，
但无法控制过滤器的执行。
由于 Spring Security 依赖 token 解析出的用户信息进行鉴权，
JWT 过滤器的执行顺序在 Spring Security 鉴权之前，
因此即使配置了白名单，所有接口依然会先经过 JWT 解析。
解决方案：
在 JWT 过滤器中特殊判断拉卡拉回调接口 /pay/lklCallback，直接跳过 token 解析逻辑。
